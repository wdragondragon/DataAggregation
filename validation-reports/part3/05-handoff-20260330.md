- 已在 `AdaptiveMergeConfig` 中补齐局部乱序与 spill 护栏配置：

- `AdaptiveMergeCoordinator` 已支持 `RECOVER_LOCAL`：
  - 不再一遇到残余乱序就直接全局 `bucketMode`
  - 改为只 spill 受污染前缀
  - 维护 `maxObservedKeyBySource`
  - 继续通过 `bucketUpperBound` 约束受污染前缀路由
- 已新增共享 `SpillGuard` 与 `SpillLimitExceededException`。
- `OverflowBucketStore`、`PartitionedSpillStore`、fusion executor、consistency executor 都已接入同一份 spill 配额控制。
- `SortMergeStats` 已补齐：
  - `localReorderedGroupCount`
  - `localMergedDuplicateGroupCount`
  - `orderRecoveryCount`
  - `spillBytes`
  - `activeSpillBytes`
  - `spillGuardTriggered`
  - `spillGuardReason`
- rebalance 子分区不再沿用父层同分区数：
  - legacy streaming 走 `cache.rebalancePartitionMultiplier`
  - adaptive sort-merge 走 `adaptiveMerge.rebalancePartitionMultiplier`
  - child 分区数按 `max(current+1, current*multiplier)` 增长
- `PartitionedSpillStore` 已支持 eager parent cleanup：
  - 父分区文件在完整消费并关闭 reader 后可立即删除
  - 删除时会同步释放 `SpillGuard` 的 active quota
  - `keepTempFiles=true` 时跳过 eager delete

## 明天接手建议

### 优先验证顺序
1. 先跑 core 层测试，确认本轮代码改动未引入基础回归。
2. 再跑 `AdaptiveSortMergeMysqlIntegrationTest`：
   - 默认小样本场景
   - 稀疏局部倒退场景
   - spill 护栏快速失败场景
3. 如果 MySQL 环境稳定，再跑 benchmark 场景观察：
  - `localReorderedGroupCount`
  - `orderRecoveryCount`
  - `spillBytes`
  - `activeSpillBytes`
  - `mergeSpilledKeyCount`

### 重点关注点
- 稀疏局部倒退场景下：
  - `executionEngine` 应保持在 `sortmerge` 或 `hybrid`
  - 不应因为一次局部倒退直接进入全局 `bucket`
  - `orderRecoveryCount` 在 source 缓冲能吸收时应保持为 0
- spill 护栏场景下：
  - consistency 应返回失败状态，并带有 `spillGuardTriggered`
  - fusion 应快速失败，并透出 `Spill limit exceeded` 之类的错误消息
- 百万级 benchmark 下：
  - 需要优先确认当前机器上的长时间运行是否来自：
    - `consistency-overflow` / `consistency-spill` 回放过慢
    - spill / rebalance 深度扩大
    - 当前参数比 part2 更早进入重 spill 路径

### 如果验证失败，优先排查
- `AdaptiveSortMergeTestSupport` 的 query 构造是否被数据源层重新排序覆盖
- `preferOrderedQuery=false` 是否在 sortmerge reader 路径里生效
- 残余乱序是否真的进入了 `RECOVER_LOCAL`，而不是被旧逻辑提前切到 `BUCKET`
- spill guard 是否在 rebalance 子层也复用了同一份配额
