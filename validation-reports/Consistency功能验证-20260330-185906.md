# Consistency功能验证

测试时间：2026-03-30 18:59:06  
测试目录：`data-mock`  
执行方式：全程通过 `JobContainer` 与 `ConsistencyExample` 交叉验证  
运行环境：`-Daggregation.home=C:\dev\ideaProject\DataAggregation\package_all\aggregation`

## 1. 测试目标

本次验证覆盖 consistency 的 3 类核心差异场景：

- `UPDATE`：目标表存在，权威表存在，但字段值不一致
- `INSERT`：目标表缺失，权威表存在
- `DELETE`：目标表存在，权威表缺失

同时做两条执行链路的交叉验证：

1. `ConsistencyExample -> DataConsistencyService -> StreamingConsistencyExecutor`
2. `JobContainer -> ConsistencyReader -> StreamingConsistencyExecutor -> mysql8 writer`

## 2. 数据模型与Mock方式

每个测试场景会动态创建 5 张表：

- `exampleTarget`
- `exampleReference`
- `readerTarget`
- `readerReference`
- `output`

其中：

- `exampleTarget/exampleReference` 用于 `ConsistencyExample` 直接执行
- `readerTarget/readerReference/output` 用于 `JobContainer + ConsistencyReader`
- `output` 表接收 `ConsistencyReader` 投影出来的差异结果

Mock 数据全部通过 `AbstractDataSourcePlugin.insertData` 初始化，字段覆盖以下基础类型：

- `BIGINT`：`biz_id`
- `VARCHAR`：`user_name`
- `INT`：`age`
- `TINYINT`：`active_flag`
- `DATE`：`register_date`
- `DATETIME`：`last_login_at`
- `TEXT`：`profile_text`、`note_text`
- `DECIMAL(18,2)`：`salary`
- `DOUBLE`：`bonus_rate`
- `CHAR(1)`：`level_code`

为避免 JDBC 返回的时间/数值类型在不同链路下形态不一致，consistency 测试源查询统一把日期、时间、数值格式化成稳定字符串后再参与比较。

## 3. 四路交叉验证内容

100 条功能验证中，同时校验以下 4 组结果：

1. `ConsistencyExample` 的 `ComparisonResult`
2. `ConsistencyExample` 执行后的目标表最终快照是否与权威表一致
3. `ConsistencyReader` 的 `ComparisonResult`
4. `ConsistencyReader` 输出表 `output` 的差异记录，以及 Reader 执行后的目标表最终快照是否与权威表一致

关键断言包括：

- 总记录数、一致数、不一致数、已解决数
- `insert/update/delete` 计数
- `output` 表中的 `conflict_type` 统计
- `output` 表中的差异 key 集合
- 最终目标表与权威表的聚合值、样本行、全量有序结果

## 4. 100条功能验证结果

执行命令：

```bash
mvn -pl data-mock -am "-Dtest=ConsistencyMysqlCrossValidationIntegrationTest" test -DfailIfNoTests=false
```

测试类：

- `data-mock/src/test/java/com/jdragon/aggregation/datamock/consistency/ConsistencyMysqlCrossValidationIntegrationTest.java`
- `data-mock/src/test/java/com/jdragon/aggregation/datamock/consistency/ConsistencyMysqlCrossValidationTestSupport.java`

验证结果：

- 场景规模：100 条
- 差异分布：25 条 `UPDATE`、25 条 `INSERT`、25 条 `DELETE`
- 预期不一致数：75
- `ConsistencyExample` 实际不一致数：75
- `ConsistencyReader` 实际不一致数：75
- `output` 表实际写出差异条数：75
- `BINARY_CONFLICT`：25
- `MISSING`：50
- `ConsistencyExample` 最终目标表与权威表全量一致
- `ConsistencyReader` 最终目标表与权威表全量一致

测试耗时：

- surefire 记录的用例耗时：`4.263s`

结论：

- `insert/update/delete` 三类差异都被正确识别
- `ConsistencyExample` 与 `ConsistencyReader` 的统计结果一致
- `Reader` 输出表与内存里的比对结果一致
- 自动修复后，目标表内容可以对齐到权威表

