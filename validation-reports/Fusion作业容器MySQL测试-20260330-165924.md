# Fusion作业容器MySQL测试-20260330-165924

## 1. 测试目标

验证 `data-mock` 模块中新建的 Fusion 测试链路是否满足以下要求：

- 使用 `FusionExample` 风格的作业配置生成方式运行。
- 强制使用 `JobContainer` 启动作业，而不是直接调用底层融合执行器。
- `aggregation.home` 固定为 `C:\dev\ideaProject\DataAggregation\package_all\aggregation`。
- MySQL 连接信息复用 `core/src/main/resources/fusion-mysql-demo.json`。
- 通过内置 `mysql8` 插件创建 2 个源表和 1 个目标表。
- 通过 `AbstractDataSourcePlugin.insertData` 初始化 mock 数据。
- 覆盖基础字段类型，并提供 100、1w、100w 三档数据量测试入口。

## 2. 测试实现概览

本次新增测试代码位于：

- `data-mock/src/test/java/com/jdragon/aggregation/datamock/fusion/FusionJobContainerMysqlTestSupport.java`
- `data-mock/src/test/java/com/jdragon/aggregation/datamock/fusion/FusionJobContainerMysqlIntegrationTest.java`

核心实现说明：

- `FusionJobContainerMysqlTestSupport` 负责：
  - 设置 `aggregation.home`
  - 从 `fusion-mysql-demo.json` 读取 MySQL 连接
  - 动态创建 3 张测试表
  - 使用 `Mysql8SourcePlugin.insertData(...)` 分批写入 mock 数据
  - 动态生成 Fusion 作业 JSON
  - 通过 `new JobContainer(Configuration.from(...)).start()` 执行
  - 回读目标表并做聚合校验
- `FusionJobContainerMysqlIntegrationTest` 负责 100 条数据的严格正确性验证。
- `FusionJobContainerMysqlBenchmarkTest` 负责 100、1w、100w 三档压测入口与报告输出。
- `FusionJobContainerMysqlAdaptiveBehaviorTest` 负责局部乱序吸收、严重乱序触发 spill、spill 清理释放配额的行为验证。

## 4. Mock 数据与融合规则

### 4.1 数据构造方式

- 每个 `biz_id` 在源表 A、源表 B 中各生成一条记录。
- 两张源表通过 `biz_id` 做 `LEFT JOIN` 融合。
- 数据通过分批 `insertData` 初始化，单批大小为 `5000`。

### 4.2 字段映射覆盖

本次测试不仅覆盖直接字段合并，也覆盖了多种 Fusion 规则类型：

- `DIRECT`
  - 例如 `sourceA.user_name -> user_name`
  - 例如 `sourceB.salary -> salary`
- `EXPRESSION`
  - `salary_double = TO_NUMBER(${sourceB.salary, Double}) * 2`
- `CONDITIONAL`
  - `age_bucket = age >= 40 ? senior : junior`
- `CONSTANT`
  - `scenario_tag = job_container`

### 5.6 局部乱序与 Spill 行为测试

执行时间：2026-03-30 17:31

执行命令：

```bash
mvn -pl data-mock -am "-Dtest=FusionJobContainerMysqlAdaptiveBehaviorTest" test -DfailIfNoTests=false
```

结果：

- 通过。
- 共覆盖 3 个自定义场景。

场景说明与验证结论：

| 场景 | 数据量 | 乱序方式 | 关键配置 | 预期 | 实际结果 |
| --- | ---: | --- | --- | --- | --- |
| `fusion_job_sparse_out_of_order` | 2000 | `sourceB` 每 128 条做一次相邻交换 | `preferOrderedQuery=false`、`pendingKeyThreshold=4096` | 稀疏乱序由 pending window 等待合并，执行引擎仍为 `sortmerge`，不触发 spill | 通过 |
| `fusion_job_spill_recovery` | 5000 | `sourceB` 整窗反转，形成明显顺序错位 | `joinType=INNER`、`pendingKeyThreshold=1` | 进入 `hybrid`，触发 spill，且最终融合结果仍正确 | 通过 |
| `fusion_job_spill_cleanup` | 8000 | `sourceB` 整窗反转，持续制造 spill 压力 | `joinType=INNER`、`pendingKeyThreshold=1`、`maxSpillBytesMB=128` | 作业完成后 `activeSpillBytes` 回到 `0`，证明 spill 文件清理后配额释放生效 | 通过 |

具体断言包括：

- `sortmerge` 场景：
  - `pendingPeakKeyCount > 0`
  - `windowImmediateResolvedKeyCount > 0`
  - `mergeSpilledKeyCount = 0`
  - `spillBytes = 0`
- `hybrid spill` 场景：
  - `executionEngine = hybrid`
  - `spillBytes > 0`
  - `mergeSpilledKeyCount > 0`
  - `windowEvictedKeyCount > 0`
- `spill cleanup` 场景：
  - `spillBytes > 0`
  - `spillLateArrivalKeyCount > 0`
  - `activeSpillBytes = 0`

补充说明：

- 测试支架为此增加了按场景控制 `join.type`、`adaptiveMerge.*` 和 source 输出顺序的能力。
- 原本尝试增加“极小 spill 配额导致作业失败”的 JobContainer 场景，但在当前实现里 spill 会边处理边释放，失败并不稳定，因此收口为“发生 spill 且 active quota 被释放”的稳定回归测试。

## 7. 结论

本次 Fusion 测试已完成以下验证：

- `JobContainer` 驱动的 Fusion 作业链路可正常运行。
- MySQL 连接可直接复用 `fusion-mysql-demo.json`。
- 通过内置 `mysql8` 插件与 `insertData` 完成了全自动 mock 初始化。
- 两源一目标的融合路径可稳定产出正确结果。
- 100 条数据完成了严格正确性校验。
- 1w 条数据完成了中量 benchmark 验证。
- 轻微局部乱序可被 source 侧缓冲吸收，不影响主路径继续走 `sortmerge`。
- 严重乱序会进入 `hybrid` 并触发 spill，且最终融合结果仍可校验正确。
- spill 场景结束后 `activeSpillBytes` 会释放回 `0`，验证了当前 eager cleanup / active quota 释放链路。
- 100w 条高压测试入口已就绪，可按需单独执行。

当前建议：

- 若下一步要评估极限吞吐和数据库压力，可单独执行 `fusion_job_large_bench`。
- 若后续需要对比不同 writer batch、大字段、局部乱序或 spill 行为，可在当前测试支架上继续扩展场景。
- 若后续要专门验证 spill guard 的失败边界，建议新增更大 payload 或更慢释放的压力模型，而不是仅靠小 `maxSpillBytesMB`。
